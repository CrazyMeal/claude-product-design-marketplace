# Architecture Handoff Template

Used by `/handoff-architecture` to produce `docs/product-handoff.md`.

---

## Output Format

```markdown
# Product Handoff — [Product/Feature Name]

> Generated by /handoff-architecture. Input for architecture agents.
> Do NOT produce architecture artifacts from this file manually.
> Use the architecture plugin commands instead: /architecture-design:design-system, /architecture-docs:tech-spec, /diagrams-as-code:c4-diagram, /architecture-docs:adr

## Problem Definition

[1–3 sentences. What problem is being solved, for whom, and why it matters.
No marketing language. Engineering-relevant framing only.]

## Users and Flows

### User Segments
[List each segment with one-line description.]

### Key User Flows
[Numbered steps for each primary flow. Focus on system interactions, not UI details.]

1. [Flow name]
   - Step 1
   - Step 2
   - ...

## Constraints

### Hard Constraints
[Non-negotiable: compliance, SLAs, existing system boundaries, data residency, etc.]

### Soft Constraints
[Preferred but flexible: tech stack preferences, team capability, timeline.]

## Edge Cases

[Specific scenarios the system must handle gracefully. Be concrete.]

- [Edge case 1]: [expected behavior]
- [Edge case 2]: [expected behavior]

## Non-Goals

[What this system explicitly does NOT need to do. Critical for scoping.]

- [Non-goal 1]
- [Non-goal 2]

## Feature Scope

### In Scope
[Numbered feature list. Prioritized — most critical first.]

1. [Feature name]: [one-line description]

### Out of Scope (for this iteration)
[Features deferred to future iterations.]

- [Deferred feature]: [reason for deferral]

## Success Metrics

[How engineering will know the system is working correctly.]

- [Metric 1]: [target value or condition]
- [Metric 2]: [target value or condition]

## Open Questions for Architecture

[Unresolved product decisions that have architectural implications. Architecture agents should surface options.]

- [Question 1]
- [Question 2]

## Next Steps

Use these architecture plugin commands to proceed:

| Command | Purpose |
|---------|---------|
| `/architecture-design:design-system [name]` | Overall system architecture — C4 diagrams, component design |
| `/architecture-docs:tech-spec [feature]` | Modular technical specification in `docs/specs/[feature]/` |
| `/diagrams-as-code:c4-diagram [system]` | Individual C4 diagram as a `.puml` file in `docs/diagrams/` |
| `/architecture-docs:adr [title]` | Architecture Decision Record in `docs/adr/` |

Start with `/architecture-design:design-system` to establish system boundaries, then `/tech-spec` for implementation details.
```

---

## Quality Checklist

Before finalizing the handoff, verify:

- [ ] Problem definition is one clear sentence — no ambiguity
- [ ] Every user flow has concrete steps, not abstract descriptions
- [ ] Hard constraints are actually hard (not preferences)
- [ ] Edge cases are specific scenarios, not categories
- [ ] Non-goals are explicit — nothing left to interpretation
- [ ] Feature scope has a priority order
- [ ] Open questions are questions, not vague concerns
- [ ] Next Steps section lists the architecture plugin commands to use
